## What We Learned

This frontend is not trying to be the full institution product yet.

It is a prototype and demo. The goal is to impress an investor or demo-day audience with one clean story:

1. an operator creates credible interview practice content
2. a student uses it in a realistic practice flow
3. the system scores the result and makes the outcome visible

The first audience to optimize for is a recruiter or career-services leader.

That means the frontend should feel like a purpose-built practice engine, not generic forms and dashboards.

## Product Thesis

Prep Dojo should feel like an operating system for interview practice.

Not a CMS.
Not a quiz app.
Not an admin portal with a student page bolted on.

The demo should make someone believe:

"This turns static prep content into a living loop of authoring, practice, scoring, and readiness."

## Existing Product Surface

The backend already gives us enough to tell a real story:

- authored topic and concept creation
- authored question create, edit, review, publish, archive
- authored question listing and filtering
- authored question submission and scoring
- practice session create, start, list, inspect, complete
- practice-session filtering by status, source, time window, current question, and remaining work
- role-aware authored routes through `X-User-Role`

What does not exist yet:

- any frontend
- full authentication UX
- production-grade identity
- analytics-heavy reporting

## The Right Problem

The problem is not "how do we build an admin panel?"

The problem is:

"How do we show that Prep Dojo can run the full interview-practice loop in a way that feels more credible and more alive than generic forms or learning dashboards?"

## Premises

1. The demo should optimize for narrative clarity over feature breadth.
2. The strongest frontend wedge is a guided end-to-end story, not a full operations suite.
3. The admin surface should be just deep enough to prove the content is real and controllable.
4. The student surface should carry most of the emotional weight of the demo.
5. The UI should feel like a purpose-built prep engine, not Airtable with nicer colors.
6. The 10x version is a personal interview coach, so even the prototype should hint at momentum, coaching, and progress.

## Experience Principles

### 1. One Story, Three Beats

The demo should be built around three moments:

1. Create
2. Practice
3. Review

Every screen should support one of those beats.

### 2. Show the Loop

The magic is not in any single screen. It is in seeing the system close the loop.

If the demo only shows authoring, it feels like a tool.
If it only shows practice, it feels like another study app.
If it shows both connected by real data, it feels like a product.

### 3. Make the Student Experience the Hero

The operator experience exists to make the student moment believable.

The student side should feel focused, calm, and high-stakes in a good way. The admin side can be sharper and denser, but it should still feel intentional.

### 4. Avoid Dashboard Bloat

No giant home page full of empty cards, tables, and placeholder charts.

For a prototype, dead UI is poison.

### 5. Favor Guided Flows Over Open Navigation

This is demo software. The fastest way to make it feel good is to control the path.

## Recommended Frontend Approach

### Approach A: Operator Console First

Summary: Build a strong content and session operations console, then add a thin student practice flow.

Effort: M
Risk: Medium
Pros:
- Easiest to map directly to the current backend endpoints
- Fast to prove content ops and lifecycle controls
- Good internal-tool foundation
Cons:
- Feels like operations software
- Weakest demo story for investors
- Student value arrives too late
Reuses:
- Authored content routes
- Practice-session list filters
- Status transition endpoints

### Approach B: Guided End-to-End Story

Summary: Build one polished three-part product story: author a question, run a student practice session, review the scored result and session state.

Effort: M
Risk: Low
Pros:
- Best demo story
- Uses real backend paths
- Keeps scope tight and finishable
- Lets design effort concentrate on the highest-value screens
Cons:
- Not a full admin product
- Some ops features stay hidden or deferred
- Requires deliberate choreography
Reuses:
- Topic, concept, and authored question routes
- Review/publish transition route
- Practice-session create, start, get, complete
- Authored submit route

### Approach C: Student Coach First

Summary: Build a strong student practice product first and treat admin as backstage tooling.

Effort: M
Risk: Medium
Pros:
- Most emotionally compelling user experience
- Strong long-term trajectory toward a personal coach
- Best chance of making someone want to use it
Cons:
- Harder to prove the content and ops system is real
- More tempting to fake backend-powered moments
- Risk of losing the full-loop story
Reuses:
- Practice-session and submit APIs
- Reference and authored retrieval flows

## Recommendation

Choose Approach B.

It is the smallest thing that can make the product feel real.

The job of this frontend is not to expose every capability in Sprint 2. The job is to make one belief obvious:

"Prep Dojo can take a question from authored content to student practice to structured review in one clean system."

That is the demo.

## Proposed Product Shape

Build a single frontend with three primary surfaces:

1. Demo landing and flow selector
2. Operator studio
3. Student practice workspace

The review result can live as the final state of the student workspace, with a linked operator-facing session detail view.

## Information Architecture

### 1. Demo Landing

Purpose:
- frame the product in one sentence
- show the three-beat flow
- let the presenter enter as operator or student

Key modules:
- headline and product promise
- "How it works" strip: Create -> Practice -> Review
- call-to-action buttons:
  - `Enter Operator Studio`
  - `Enter Student Practice`
  - `View Demo Session`

### 2. Operator Studio

Purpose:
- create and publish one believable piece of content
- prove that content is structured, reviewable, and controlled

Core screens:

#### Operator Home

Not a dashboard. More like a command surface.

Sections:
- draft content in progress
- ready to publish
- recent sessions needing attention
- quick action to create a new question

#### Question Composer

Single focused creation flow:
- topic
- concept
- assessment mode
- prompt
- rubric
- expected answer
- common mistakes

The important part is making the rubric visible and legible. This is where credibility comes from.

#### Review and Publish Panel

One clean step where the operator:
- sees content summary
- adds review notes
- moves `draft -> reviewed -> published`

This should feel like content quality control, not just a status dropdown.

#### Session Triage View

Small but real.

Show:
- session status
- source
- current question
- remaining count
- started/completed timing

This is enough to prove the platform is operational.

### 3. Student Practice Workspace

Purpose:
- deliver the most impressive part of the product

Core states:

#### Session Ready

Show:
- session title
- question count
- first question preview
- strong primary CTA: `Start Practice`

#### Active Question

Show:
- question prompt
- answer input
- session progress
- subtle timer or focus treatment
- submit CTA

This page should feel serious, not playful. Closer to interview prep than gamified edtech.

#### Result View

Show:
- overall score
- mastery band
- criterion-level feedback
- strengths
- misses
- next-step prompt

This is the payoff screen. It needs to feel sharp.

#### Session Complete

Show:
- completed state
- what was practiced
- result summary
- invitation to continue or review

## Demo Script

This should be the golden path:

1. Start on the landing page.
2. Enter Operator Studio.
3. Create or open a finance interview question.
4. Review and publish it.
5. Switch to Student Practice.
6. Start a practice session with that question.
7. Submit an answer.
8. Show score and feedback.
9. Return to the operator view and show the session now appears in the system with updated state.

If this flow is smooth, the product feels real.

## Visual Direction

The UI should feel more like a high-trust prep product than enterprise software.

Direction:
- editorial typography, not default SaaS fonts
- warm neutrals with one confident accent color
- restrained motion, especially when moving from authoring to practice to review
- large, readable question prompts
- rubric and score presentation that feels analytical, not cluttered

Avoid:
- generic KPI cards
- empty charts
- loud gamification
- purple-on-white startup gradients
- giant sidebars with ten dead nav items

## Suggested Screen Set

Keep V1 to 6 screens:

1. Landing / story page
2. Operator home
3. Question composer
4. Review and publish screen
5. Student practice workspace
6. Result and session review screen

Anything beyond that is probably ocean, not lake.

## Backend Mapping

The frontend should use real backend flows wherever possible.

### Operator Flows

- `POST /api/v1/authored/topics`
- `POST /api/v1/authored/concepts`
- `POST /api/v1/authored/questions`
- `GET /api/v1/authored/questions`
- `GET /api/v1/authored/questions/{question_id}`
- `PUT /api/v1/authored/questions/{question_id}`
- `POST /api/v1/authored/questions/{question_id}/status`
- `GET /api/v1/practice-sessions`

Use role header:
- `X-User-Role: academic`

### Student Flows

- `POST /api/v1/practice-sessions`
- `POST /api/v1/practice-sessions/{session_id}/start`
- `POST /api/v1/authored/questions/{question_id}/submit`
- `GET /api/v1/practice-sessions/{session_id}`
- `POST /api/v1/practice-sessions/{session_id}/complete`

No real login UX is needed for the prototype. Role and persona can be chosen in the frontend shell.

## What To Fake, What To Keep Real

Keep real:
- content creation
- publish transition
- session creation
- answer submission
- score and feedback rendering
- session state changes

Safe to fake or simplify:
- full authentication
- multi-user switching
- broad analytics
- full topic/concept management depth
- archive flows
- every filter surface

This is important. A prototype gets stronger when it fakes the boring edges and keeps the magic real.

## Risks

### Risk 1: Building a generic admin app

If the operator side dominates, the product feels procedural.

Mitigation:
- keep the admin surface narrow
- spend design effort on the student and result views

### Risk 2: Building too many screens

This turns into UI sprawl fast.

Mitigation:
- cap V1 at six screens
- design the demo path first, then optional navigation

### Risk 3: Overexposing backend complexity

Lifecycle states and filters are useful, but not all of them belong in the first demo.

Mitigation:
- show only the controls needed for the story

## Build Sequence

### Slice 1: Story Shell

- landing page
- role/persona switcher
- top-level layout and design system primitives

### Slice 2: Operator Story

- operator home
- question composer
- review/publish screen

### Slice 3: Student Story

- session ready
- active question
- submit flow
- result view

### Slice 4: Loop Closure

- session detail review
- operator visibility of the just-completed session
- final polish for transitions and copy

## Success Criteria

The frontend prototype is successful if:

1. someone watching the demo understands the product in under 60 seconds
2. the student practice flow feels like the center of gravity
3. the operator flow proves the system is controllable and credible
4. the result screen makes the feedback loop feel real
5. the product does not feel like generic dashboard software

## Final Recommendation

Do not start by building a full admin console.

Build the guided story first:

1. operator creates and publishes a question
2. student practices it
3. system returns structured feedback
4. operator sees the session state update

That is enough to make the product feel inevitable.
